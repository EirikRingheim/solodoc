# Module Spec: Multi-Language Support

## Overview
Solodoc supports full multi-language functionality from day one. Foreign workers
in Norwegian construction, farming, and machine operation must be able to understand
safety-critical information in their own language. The system uses a combination of
professional UI translations, AI/machine translation for dynamic content, and
language-independent visual elements (GHS pictograms, PPE icons).

Every piece of translated content shows a clear "automatically translated" indicator
so users know to verify anything that seems incorrect.

| Language | Code | Primary Use Case |
|----------|------|-----------------|
| Norwegian Bokmål | nb-NO | Default language, all admin content created in Norwegian |
| English | en | Universal second language, international workers |
| Polish | pl | Largest foreign workforce in Norwegian construction |
| Lithuanian | lt | Significant Baltic workforce in construction |
| Romanian | ro | Growing presence in construction sector |
| Spanish | es | South American workforce |

### Adding More Languages
- The architecture supports any language DeepL can translate
- Adding a new language is a configuration change, not a code change
- DeepL supports 30+ languages — future additions are trivial
- New languages are added by Solodoc staff and become available to all tenants

---

## Language Selection

### User Level
- Each user selects their preferred language in their profile: Min profil → Språk
- Default: Norwegian (nb-NO)
- The entire UI and all translated content appears in their selected language
- Language preference is stored on the user account and applies across all devices

### Tenant Level (Admin Controls)
- Admin selects which languages are available for their tenant:
  Innstillinger → Språk → "Aktive språk"
- Only enabled languages appear in the user language picker
- Default: Norwegian + English enabled. Others disabled until admin enables them.
- Enabling a language triggers background translation of all existing templates
  and procedures (queued as a background job)
- Admin can disable a language they no longer need (translations are preserved
  but the language disappears from the picker)

### Why Admin Controls Languages
- A small farming operation with only Norwegian workers shouldn't pay for
  or be confused by 6 language options
- A large construction firm with Polish and Lithuanian crews enables those languages
- Translation costs (DeepL API calls) are only incurred for languages the tenant uses

## Translation Architecture

### Three Categories of Content

**Category 1: Solodoc UI (Static)**
- Buttons, labels, navigation, error messages, system notifications
- Translated by Solodoc team (professional translation, not AI)
- Stored as resource files using IStringLocalizer (.resx or JSON)
- These must be perfect — no machine translation for core UI
- Updated with each Solodoc release

**Category 2: Dynamic Content (AI/Machine Translated)**
- Admin-created content: checklist templates, procedures, announcements,
  project descriptions, checklist item labels, procedure text blocks
- Translated automatically using DeepL API
- Cached after translation — not regenerated on every view
- Retranslated when the original Norwegian content is edited
- Marked with "automatically translated" indicator

**Category 3: Safety Content (Hybrid)**
- GHS H-statements and P-statements: use official EU translations
  (these exist in all EU languages as part of CLP regulation — not AI translated)
- SDS AI summaries: generated by Anthropic API in the target language directly
  (not translated from Norwegian — generated natively for better quality)
- First aid instructions: from official SDS translations where available,
  DeepL for the rest
- PPE requirements: short phrases, DeepL translated + cached

**Category 4: Language-Independent (No Translation Needed)**
- GHS pictograms (universal symbols)
- PPE icons (visual, not text)
- Numbers, dates, measurements
- Photos and attachments
- Signature images
- Barcodes and QR codes

## Translation Engine

### Primary: DeepL API
- Used for all dynamic content translation (Category 2)
- Best-in-class quality for European languages, especially Norwegian → Polish/Lithuanian/Romanian
- REST API: `https://api-free.deepl.com/v2/translate` (free tier) or
  `https://api.deepl.com/v2/translate` (Pro)
- Free tier: 500,000 characters/month (likely sufficient for small-medium tenants)
- Pro tier: €5.49/month for 1,000,000 characters (for high-volume tenants)
- Supports glossaries for domain-specific terminology (e.g., ensure "avvik"
  consistently translates to the correct construction term in each language)

### Secondary: Anthropic API
- Used only for generating SDS AI summaries in multiple languages
- The summary is generated natively in the target language (not translated)
- This produces better results for safety content because the AI understands
  the context, not just the words
- More expensive per call — reserved for safety-critical content only

### Service Abstraction
- Translation is behind `ITranslationService` interface
- DeepL implementation is the default
- Can be swapped to Google Cloud Translation or another provider without code changes
- All translation calls are async and non-blocking

## Translation Workflow

### When a Template is Created/Edited
1. Admin creates or edits a checklist template in Norwegian
2. On save, a background job is queued for each enabled language
3. DeepL translates: template name, description, all item labels, help text,
   section headers
4. Translations are stored in a `Translation` table:
   `EntityType, EntityId, FieldName, LanguageCode, TranslatedText, TranslatedAt`
5. When a Polish worker views the template, the Polish translations are loaded
6. If translation fails (API error), the Norwegian original is shown with a note:
   "Oversettelse ikke tilgjengelig — viser norsk tekst"

### When an Announcement is Posted
1. Admin writes announcement in Norwegian
2. Background job translates to all enabled languages
3. Each user sees the announcement in their preferred language
4. Translation happens asynchronously — there may be a brief delay (seconds)
   between posting and translations being available
5. If a user opens the announcement before translation completes,
   they see the Norwegian original

### When a Chemical is Added
1. Admin adds chemical, AI summary is generated in Norwegian
2. Background job generates AI summaries in each enabled language
   (via Anthropic API — native generation, not translation)
3. Official H/P-statements are loaded from the EU translation database
   for each enabled language
4. PPE requirement text is translated via DeepL
5. All translations cached on the chemical record

## "Automatically Translated" Indicator

### What It Looks Like
- Small, non-intrusive banner at the top of any translated content:
  "⚡ Automatisk oversatt — kontroller viktig informasjon"
  (shown in the user's selected language)
- Does NOT appear on:
  - Solodoc UI elements (these are professionally translated)
  - Official H/P-statements (these are official EU translations)
  - GHS pictograms and icons (language-independent)
- DOES appear on:
  - Translated checklist templates and items
  - Translated procedures
  - Translated announcements
  - AI-generated SDS summaries in non-Norwegian languages
  - Translated first aid and PPE text

### Why This Matters
- Builds trust: users know the system is being transparent
- Safety net: if a translation is wrong for safety-critical content,
  the user knows to verify
- Legal protection: Solodoc is clear that translations are automated

### Purpose
Construction and safety terminology must translate consistently. DeepL supports
custom glossaries that enforce specific translations for domain terms.

### Example Glossary Entries

### Glossary Management
- Solodoc maintains the master glossary (updated with each release)
- Glossary is uploaded to DeepL via their Glossary API
- Ensures all tenants get consistent terminology
- Tenants cannot modify the glossary (consistency across the platform)

## Offline Language Support

### What's Available Offline
- Solodoc UI strings for the user's selected language (bundled in PWA)
- Cached translations of templates and checklist items
- Chemical AI summaries in the user's language
- GHS pictograms and PPE icons (always available)
- H/P-statements in the user's language

### What Requires Connectivity
- Translating new content (DeepL API call needed)
- Generating new AI summaries (Anthropic API call needed)
- Loading translations that haven't been cached yet

## Cost Management

### Translation Costs Per Tenant
- DeepL free tier: 500,000 chars/month — sufficient for small tenants
- DeepL Pro: €5.49/month — sufficient for most medium tenants
- Anthropic API for SDS summaries: minimal (only on chemical register setup)
- Cost scales with: number of enabled languages × amount of content created

### Optimization
- Translate only when content changes (not on every view)
- Cache aggressively — translations are stored in database
- Batch translation requests where possible (DeepL supports arrays)
- H/P-statements are pre-translated (no API cost — official EU translations)
- GHS pictograms have zero translation cost

### Billing Consideration
- Translation costs are absorbed into the Solodoc subscription
  (not billed separately to tenants)
- If a tenant enables all 6 languages and creates hundreds of templates,
  costs increase — this should be factored into pricing tiers
- Alternatively: include 2 languages in base price, charge per additional language
  (decision deferred to pricing strategy)
